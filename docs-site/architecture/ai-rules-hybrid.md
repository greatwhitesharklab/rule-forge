# AI+规则混合架构

RuleForge 的核心设计理念：**规则是护栏，AI 是引擎**。

## 为什么需要混合？

| 纯规则 | 纯 AI | AI+规则混合 |
|-------|-------|------------|
| ✅ 可审计 | ❌ 黑盒 | ✅ 可审计 |
| ✅ 可解释 | ❌ 难解释 | ✅ 可解释 |
| ❌ 适应性差 | ✅ 自适应 | ✅ 自适应 |
| ❌ 维护成本高 | ❌ 不可控 | ✅ 可控 |

在金融场景中，**可审计和可解释是不可妥协的合规要求**。AI+规则混合模式在保持合规性的同时引入 AI 的智能能力。

## 架构设计

```
                        ┌─────────────────┐
                        │   业务请求       │
                        └────────┬────────┘
                                 │
                        ┌────────▼────────┐
                        │  ML 模型推理     │  ← Python Model Service
                        │  (欺诈概率/评分)  │     (PKL/PMML)
                        └────────┬────────┘
                                 │ probability
                        ┌────────▼────────┐
                        │  规则引擎决策     │  ← RETE Engine
                        │  (阈值+业务规则)  │     (确定性护栏)
                        └────────┬────────┘
                                 │ decision
                        ┌────────▼────────┐
                        │   审计日志       │  ← 可追溯
                        └─────────────────┘
```

## ML 模型集成

### Model Service

独立的 Python FastAPI 服务，负责加载和管理 ML 模型：

- 支持 PKL 格式（scikit-learn）
- 自动检测模型输入字段
- REST API 接口
- 模型热加载

### PklModelConnector

Java 端连接器，在规则引擎中调用 Model Service：

1. 规则流执行到「模型预测」节点
2. PklModelConnector 收集输入数据
3. 通过 HTTP 调用 Model Service
4. 获取预测结果（如欺诈概率）
5. 将结果作为变量传入后续规则
